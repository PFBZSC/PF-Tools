# 模块化 CLI 工具 `pt` 的子模块开发规范

我现在有一个基于 Python 的模块化命令行工具，主命令为 `pt`，采用动态路由加载。请基于以下规范为我编写一个新的子功能模块：

## 1. 命名与结构规范
- 模块必须是单文件 Python 脚本，存放在 `commands/` 目录下。文件名即指令原名。
- 文件顶部必须定义全局字符串变量 `__description__`，简要描述该模块功能。
- 文件顶部必须定义全局列表变量 `__aliases__`，提供 1~2 个常用的短指令别名（例如 `["ext", "e"]`），若无别名则为空列表 `[]`。
- 可选元数据：如需记录模块的额外信息，可在文件顶部定义双下划线的全局字符串变量（如 __author__, __version__, __github__, __email__ 等）。这些字段是完全可选的，主程序会自动识别并读取它们。

## 2. 接口规范
- 必须实现核心入口函数 `def execute(args: list[str]) -> int:`。
- `args` 是一个列表，包含了该子指令之后的所有命令行参数（已自动剔除了 `pt` 和指令名本身）。
- `execute` 函数执行完毕后，必须返回一个整数作为 Exit Code（0 成功，非 0 失败）。

## 3. 帮助与参数解析规范
- 模块必须具备自我解释能力。在 `execute` 内部应优先判断 `args` 是否为空，或者是否包含 `-h`、`--help`。
- 【关键】：如果是请求帮助，需打印详细用法并 `return 0`。打印用法的标准格式必须始终以 `pt` 开头，例如：`用法: pt <指令名或别名> [参数]`。
- 若参数错误，应打印错误提示和用法，并 `return 1`。

## 4. 异常处理规范
- 核心业务包裹在 `try...except` 块中，防止崩溃抛出 traceback，捕获异常后打印简洁信息并 `return 1`。

## 5. 跨模块调用文档输出规范
- 每次编写完功能代码后，请务必在回答的最后，附带输出一份精简的 Markdown 接口文档（格式参考下方模板）。
- 这份文档的作用是为后续其他 AI 开发新模块时，提供跨模块调用（Import）的参考依据。

请按照以下模板输出说明文档：
